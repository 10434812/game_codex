# 游戏流程接通设计

**日期：** 2026-04-16  
**项目：** `/Users/hh/Desktop/game_codex`  
**范围：** 在纯前端微信小程序中接通首页、房间页、对局页、结算页的完整游戏流程，不改动任何前端页面结构与样式。

## 1. 目标

在不改动现有 `wxml` 和 `wxss` 的前提下，将项目从静态演示页面改造成可运行的单局游戏流程：

1. 用户从首页进入房间。
2. 房间支持 4-10 人。
3. 房间真人不足时自动补机器人，确保可以开局。
4. 开局后按两人一队进行“加减分联动”。
5. 对局时长固定 180 秒。
6. 时间结束后自动进入结算页。
7. 结算页展示真实的前三名和 4-10 名排名。

## 2. 非目标

以下内容不在本次范围内：

1. 不接真实后端服务。
2. 不接多人实时联网。
3. 不改任何页面布局、样式、文案结构。
4. 不增加新的页面。
5. 不引入重型状态管理库。

## 3. 硬约束

### 3.1 页面约束

必须保留现有页面结构，不修改：

1. `pages/home/index.wxml`
2. `pages/home/index.wxss`
3. `pages/room/index.wxml`
4. `pages/room/index.wxss`
5. `pages/arena/index.wxml`
6. `pages/arena/index.wxss`
7. `pages/result/index.wxml`
8. `pages/result/index.wxss`

允许修改各页面的 `index.js`，但只能通过已有字段把数据接入页面。

### 3.2 游戏规则约束

1. 房间最少 4 人，最多 10 人。
2. 游戏时长固定 180 秒。
3. 采用两人一队。
4. 页面上仍展示个人分数，不新增队伍展示 UI。
5. 队伍效果体现在加减分联动上，而不是独立的队伍分面板。

## 4. 当前项目现状

当前项目只有四个页面，且全部使用静态假数据：

1. `home` 页面可以触发开始。
2. `room` 页面可以展示固定槽位并开始游戏。
3. `arena` 页面可以展示玩家节点和时间区块。
4. `result` 页面可以展示前三和详细排名。

问题在于：

1. 页面之间没有共享状态。
2. 房间成员、头像、分数、排名都不是同一局数据。
3. 没有房间人数规则。
4. 没有机器人逻辑。
5. 没有对局计时。
6. 没有真实结算。

## 5. 目标方案

采用“独立游戏状态模块 + 纯规则引擎”的方式接通整个项目。

### 5.1 模块划分

新增三个逻辑模块：

1. `utils/constants.js`
   - 保存人数、时间、积分区间、机器人名池等常量。
2. `utils/game-engine.js`
   - 保存纯规则逻辑，例如创建玩家、补机器人、组队、单回合积分变化、排行计算。
3. `utils/game-store.js`
   - 保存当前局状态，并提供对外动作，例如创建房间、开始游戏、推进时间、完成结算、重开一局。

页面 JS 只做两件事：

1. 读取 store 的当前状态并映射到页面现有 `data`。
2. 调用 store 动作并跳转页面。

这样可以保证页面不变，规则集中，后续如果要接真实后端，也只需要替换 store 或 engine 层。

## 6. 状态模型设计

### 6.1 全局游戏状态

全局状态结构定义为单一对象，放在 `game-store` 中统一维护：

```js
{
  roomId: '8888',
  stage: { id, code, name, image },
  status: 'idle' | 'room' | 'playing' | 'finished',
  duration: 180,
  timeLeft: 180,
  round: 0,
  players: [
    {
      id,
      name,
      avatar,
      isRobot,
      isSelf,
      seat,
      score,
      teamId,
      ready,
      state
    }
  ],
  teams: [
    {
      id,
      memberIds: ['p1', 'p2']
    }
  ],
  result: {
    ranking: [],
    top3: null,
    rest: []
  }
}
```

### 6.2 玩家模型

每个玩家必须具备以下字段：

1. `id`：唯一标识。
2. `name`：昵称。
3. `avatar`：头像路径。
4. `isRobot`：是否机器人。
5. `isSelf`：是否当前用户。
6. `seat`：房间位置，用于房间页和战场页的顺序稳定。
7. `score`：当前分数，最小值为 `0`。
8. `teamId`：所属队伍。
9. `ready`：房间准备状态。
10. `state`：房间页展示用的状态文案。

### 6.3 队伍模型

队伍只作为规则载体，不要求页面额外展示。每队固定两人：

```js
{
  id: 'team-1',
  memberIds: ['player-1', 'player-2']
}
```

## 7. 房间规则

### 7.1 创建房间

用户从首页点击“快速开始”后：

1. 初始化当前局。
2. 根据首页当前选中的景区写入 `stage`。
3. 默认创建一个包含当前用户的房间。
4. 若当前房间人数少于 4，则补机器人到 4 人。
5. 跳转到 `room` 页面展示房间。

### 7.2 房间人数

1. 房间总人数下限为 4。
2. 房间总人数上限为 10。
3. 默认单机场景下不会有真实其他玩家，因此主要通过机器人补位。
4. 若未来加入真人加入逻辑，仍然沿用同一套人数上限和 seat 规则。

### 7.3 房间槽位映射

`room` 页当前有 9 个格子，因此映射策略如下：

1. 展示时先按 seat 升序取前 9 个位置。
2. 如果当前房间有 10 人，第 10 人不在房间页展示，但会出现在战场和结算中。
3. 当前版本默认房间页实际人数控制在 4-9 人，避免房间页与战场页人数显示口径不一致。
4. 战场页和结算页允许完整展示最多 10 人。

实现上保留最大 10 人规则，但默认开局人数取 `4-9` 之间的随机值，更符合当前 UI 容量。

## 8. 组队规则

### 8.1 组队方式

1. 开局前按玩家 seat 顺序两两组队。
2. 第 1、2 位组成第 1 队，第 3、4 位组成第 2 队，依次类推。
3. 若人数是奇数，则先补一个机器人，保证所有队伍都是双人队。

### 8.2 分数展示

1. 页面仍显示个人分数。
2. 不新增队伍积分 UI。
3. 结算排名按个人分数排序。
4. 队伍只影响每回合的分数联动。

### 8.3 联动规则

每回合选中一个玩家作为事件主执行者后：

1. 如果主执行者获得正分，队友获得主执行者分值的 `50%`，向下取整，最少 `1` 分。
2. 如果主执行者扣分，队友同步承担主执行者扣分值的 `50%`，向下取整。
3. 主执行者和队友都不能低于 `0` 分。
4. 队友不触发二次连锁，不会把分数继续传导到其他人。

示例：

1. 玩家 A `+60`，其队友 B `+30`。
2. 玩家 C `-35`，其队友 D `-17`。

## 9. 对局规则

### 9.1 时长

1. 对局总时长固定 180 秒。
2. `arena` 页面倒计时显示真实剩余秒数。
3. 时间归零后自动结算并跳转 `result` 页面。

### 9.2 回合节奏

为了在现有页面中看到分数变化，对局按固定 tick 推进：

1. 每 `3` 秒触发一轮积分事件。
2. 每轮从所有玩家中随机挑选 `1-3` 个不同玩家触发事件。
3. 每次事件独立计算个人加减分和队友联动。

这样 180 秒一共约 60 轮，足够让页面分数有明显变化。

### 9.3 事件概率

为了让结果既随机又不完全失控，事件规则如下：

1. 正分概率 `60%`。
2. 负分概率 `40%`。
3. 正分区间 `+20` 到 `+80`。
4. 负分区间 `-10` 到 `-60`。
5. 当前用户与其队友的正分概率可增加一个小幅偏置，但不能高到保送第一。

偏置策略：

1. 当前用户触发正分概率 `65%`。
2. 当前用户队友触发正分概率 `62%`。
3. 其他玩家维持 `60%`。

## 10. 页面接入设计

### 10.1 首页 `pages/home/index.js`

职责调整：

1. 保留现有导航和景区切换逻辑。
2. 点击“快速开始”时，不再直接进入静态战场。
3. 改为调用 `gameStore.createRoomFromStage(activeStage)`。
4. 创建房间后跳转 `room` 页面。

### 10.2 房间页 `pages/room/index.js`

职责调整：

1. 页面加载时读取当前房间状态。
2. 将真实玩家映射到现有 `slots` 数据结构。
3. 机器人显示为已填充玩家。
4. 未使用槽位显示“等待加入”。
5. 点击开始游戏时调用 `gameStore.startGame()`。
6. 若人数不足 4，先补机器人到 4，再开局。
7. 开局后跳转 `arena` 页面。

### 10.3 对局页 `pages/arena/index.js`

职责调整：

1. 页面加载和显示时读取当前对局状态。
2. 复用现有圆盘节点布局，只替换玩家数据来源。
3. `timer` 区块显示真实 `timeLeft`。
4. 页面打开后开始订阅或轮询 store 状态刷新。
5. 对局进行中每秒刷新一次页面数据。
6. 时间结束时自动跳转 `result` 页面。
7. 页面卸载时清理本页定时器和监听，避免重复推进。

### 10.4 结果页 `pages/result/index.js`

职责调整：

1. 从 store 中读取最终排名结果。
2. 用前 3 名填充 `top3`。
3. 用第 4 到第 10 名填充 `rankList`。
4. “再来一局”调用 `gameStore.restartGame()`。
5. “返回首页”调用 `gameStore.resetToHome()` 后回首页。

## 11. 排名与结算规则

### 11.1 排名生成

1. 按最终 `score` 倒序排序。
2. 分数相同则按 seat 升序，保证排序稳定。
3. 取前 3 名作为 podium。
4. 取第 4-10 名作为详细排名列表。

### 11.2 结果页字段映射

1. `top3.first` 对应第 1 名。
2. `top3.second` 对应第 2 名。
3. `top3.third` 对应第 3 名。
4. `rankList` 对应第 4 名及以后。

### 11.3 收益区块

结果页已有“本局收益”和“新成就”区域，但当前没有规则来源。本次使用纯前端可计算值填充：

1. 经验值 = 当前用户最终分数与初始分数的正向差值，至少显示 `0`。
2. 金币 = 经验值乘以一个固定系数，例如 `3`。
3. 成就根据当前景区名称生成，例如 `长城守望者`、`富士远眺者`。

## 12. 容错与生命周期

### 12.1 页面切换

1. 从房间页返回再进入，不应重复创建房间。
2. 从对局页切走再回来，应继续当前局，而不是重开。
3. 若已经结算，进入 `arena` 应自动跳到 `result`。

### 12.2 定时器安全

1. 游戏推进定时器只能在 store 内存在一份。
2. 页面自身只负责刷新显示，不负责推进游戏主时间轴。
3. 防止 `onLoad`、`onShow` 多次触发导致重复计时。

### 12.3 空状态兜底

1. `room` 页面没有房间数据时，自动按默认景区创建新房间。
2. `arena` 页面没有对局数据时，自动回房间或重建一局。
3. `result` 页面没有结算结果时，自动跳首页。

## 13. 实现细节约束

### 13.1 数据持久化

本次不要求跨应用重启持久化，默认用内存态即可。

可选兜底：

1. 在 `App.globalData` 中保存 `gameStore` 当前状态引用。
2. 必要时使用 `wx.setStorageSync` 保存最近一局快照，但不作为主要状态源。

当前推荐：以内存 store 为主，不增加 storage 复杂度。

### 13.2 可测试性

纯规则逻辑必须从页面文件中拆出，放入 `game-engine.js`。这样可以直接在 Node 环境中验证：

1. 房间补人规则。
2. 最大最小人数规则。
3. 奇数人数补机器人组队规则。
4. 加减分联动规则。
5. 排名与结果生成规则。

### 13.3 与未来后端兼容

为了后续真接后端，页面层不能直接拼业务规则。必须满足：

1. 页面只读 `gameStore` 提供的数据。
2. 页面动作只调用 `gameStore` 暴露的方法。
3. `game-engine` 只做纯函数计算。

## 14. 验证计划

实现完成后必须至少验证以下内容：

1. 首页点击“快速开始”能进入房间。
2. 房间页能展示真实成员，不足时有机器人。
3. 房间人数不会小于 4，不会大于 10。
4. 开局后 `arena` 页面分数会实时变化。
5. 对局倒计时从 180 秒递减到 0。
6. 时间到后自动跳转结果页。
7. 结果页前三和详细排名来自同一局真实结果。
8. “再来一局”会重置并启动新的一局。
9. “返回首页”会清理当前局状态。

## 15. 风险与处理

### 15.1 房间页只有 9 个槽位

风险：规则允许 10 人，但房间页现有 UI 只有 9 格。  
处理：逻辑层保留 10 人上限，但默认单机场景下开局人数控制在 4-9；如果后续必须显示 10 人，需要另开页面改版，不在本次范围内。

### 15.2 页面无队伍展示区

风险：用户看不到组队关系。  
处理：本次只在规则层体现队伍联动，页面保持现状，不新增 UI。

### 15.3 小程序页面生命周期复杂

风险：重复进入页面可能导致多重定时器。  
处理：统一由 `gameStore` 托管唯一推进器，页面只订阅展示。

## 16. 最终结论

本次采用“页面不动、状态与规则外置”的方案，将现有静态前端接成一个纯前端可运行的单局游戏：

1. 首页建房。
2. 房间页补机器人并开局。
3. 战场页跑满 180 秒倒计时和随机积分联动。
4. 结果页展示真实结算。

该方案满足当前硬约束，且为以后接后端保留了清晰的替换边界。
